# 01. ReplicationGraph 核心概念

## 为什么需要 ReplicationGraph

默认 UE 复制路径中，每个 `UNetConnection` 往往需要对大量 Actor 做相关性判断（如 `IsNetRelevantFor`）。当 **Actor 数量 × 连接数** 增大时，CPU 开销会明显上升。

ReplicationGraph 的核心思路：

> 用一张**持久的复制图**，把 Actor 预先归类到各类 **Node**；每个连接复制时，从图中 **Gather** 列表，而不是对每个 Actor 重复做全量相关性扫描。

引擎头文件注释（`ReplicationGraph.h`）概括了收益：**更多工作可共享**，扩展性更好。

## 心智模型

```text
                    UReplicationGraph（子类实现）
                              |
        +---------------------+---------------------+
        |                     |                     |
   Global Nodes          Shared Nodes         Per-Connection Nodes
   (所有连接共用)         (如按队伍共享)          (如 AlwaysRelevant_ForConnection)
        |                     |                     |
   GridSpatialization    AlwaysRelevantTeam    AlwaysRelevant_ForConnection
        |                     |                     |
        +---------------------+---------------------+
                              |
              每个连接：GatherActorListsForConnection
                              |
              距离/频率剔除 → 合并排序 → 更新 Actor Channel
```

## 三个可扩展层

游戏代码主要通过以下三层影响复制行为（替代或弱化 `IsNetRelevantFor` / `GetNetPriority`）：

| 层级 | 类型 | 作用 |
|------|------|------|
| 图结构 | `UReplicationGraph` 子类 | 创建 Node、路由 Actor 进出 Node |
| 全局 Actor 元数据 | `FGlobalActorReplicationInfo` | 每个 Actor 在图上的全局策略（距离、频率、标志等） |
| 每连接 Actor 元数据 | `FConnectionReplicationActorInfo` | 某连接视角下该 Actor 的状态 |

## 子类必须实现的入口

`UReplicationGraph` 为抽象基类，**默认实现不能单独工作**。子类通常重写：

| 函数 | 职责 |
|------|------|
| `InitGlobalActorClassSettings` | 按 Actor 类初始化 `GlobalActorReplicationInfoMap` |
| `InitGlobalGraphNodes` | 创建全局 Node（`CreateNewNode` + `AddGlobalGraphNode`） |
| `InitConnectionGraphNodes` | 为每个连接创建或绑定 Node |
| `RouteAddNetworkActorToNodes` | Actor 开始复制时，放入对应 Node |
| `RouteRemoveNetworkActorToNodes` | Actor 停止复制时，从 Node 移除 |

Actor 生成/销毁时，引擎会调用 Route* 系列，由你的图决定「这个 Actor 进哪个桶」。

## 常见 Node 类型（概念）

引擎内置多种 Node，实践中常见：

- **GridSpatialization2D**：按网格做距离相关（大世界 / 平面关卡）
- **ActorList**：简单列表，配合策略使用
- **AlwaysRelevant**：对所有连接始终相关
- **AlwaysRelevant_ForConnection**：仅对特定连接始终相关（如 Owner、观战）
- **AlwaysRelevant_ForTeam**：对同一队伍连接相关（FPS 类项目常见）

`UBasicReplicationGraph` 只用了其中一小部分，适合作为第一个可读示例。

## 与默认复制的主要差异

| 默认路径 | ReplicationGraph |
|----------|------------------|
| 常依赖 `IsNetRelevantFor` | 由 Node + 全局/连接元数据驱动 |
| 每连接扫描成本高 | 列表持久化，Gather 成本更可控 |
| 配置分散在 Actor 上 | 策略集中在 Graph 子类，便于调优 |
| 上手简单 | 需要 C++ 子类 + ini 指定 `ReplicationDriverClassName` |

注意：从 Graph 拿到列表之后，引擎仍会做距离、频率等二次剔除与优先级排序，并不是「进了 Node 就一定每帧全量同步」。

## 启用方式（概念）

在 `DefaultEngine.ini`（或对应 NetDriver 配置）中指定复制驱动类，例如引擎示例：

```ini
[/Script/OnlineSubsystemUtils.IpNetDriver]
ReplicationDriverClassName="/Script/ReplicationGraph.BasicReplicationGraph"
```

项目实践时应改为自己的子类，例如：

```ini
ReplicationDriverClassName="/Script/DragonOathNet.DragonOathReplicationGraph"
```

同时需在 `.uproject` 中启用 **ReplicationGraph** 插件，并在游戏模块中依赖 `ReplicationGraph` 模块。

## 对 DragonOath 的启示

横版 2.5D 联机特点：

- 相关性强依赖 **横向视野** 与 **房间/段落**，而不一定是全 3D 球体距离
- 玩家、召唤物、弹幕、Boss 的复制频率与相关性差异大
- 队友协作可能需要「队伍始终相关」类 Node

因此长期目标不是直接照搬 `BasicReplicationGraph`，而是：

1. 先跑通 Basic / 最小子类；
2. 再为横版场景定制 Grid 或自定义 Node；
3. 把投射物、Buff 表现等与 GAS 复制策略对齐。
