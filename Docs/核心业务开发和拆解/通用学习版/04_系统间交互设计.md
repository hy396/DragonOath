# 通用学习版 · 系统间交互设计

> 与引擎无关。讲清三系统怎么协作、用什么方式解耦、服务端权威怎么落地。

## 1. 一张图（含服务端权威）
```text
                ┌─────────────── 服务端 / 权威端 ───────────────┐
                │ 引导配置+进度持久   红点数量+已读库   日志上报接收 │
                └───────┬──────────────────┬──────────────────┬───┘
           拉配置/同步   │         下发数量/推送变更   │        接收错误上报
                        ▼                  ▼                  ▼
业务系统（技能/邮件/任务/输入）
   │ SetActive/SetCount/markDirty    │ 发完成事件
   ▼                                ▼
红点系统  ←──── 联动 ────→  引导系统
   │ 亮/灭通知                       │ 步骤推进通知
   └──────────────┐        ┌──────────────┘
                  ▼        ▼
              事件总线（Event Bus / 消息总线）
                  │
                  ▼
              UI（只听、只显示）

所有系统 ──写日志──▶ 日志系统 ──▶ 本地文件 + 内存缓冲 + 崩溃抓栈 ──错误上报──▶ 服务端
```

## 2. 三者如何联动（最典型的几条线）
### 线 A：业务 → 红点 → UI
```text
技能可升级 → 红点.SetCount("skill", 1)
           → 脏刷新 → 父菜单自动聚合出亮
           → 发 "reddot.changed" → UI 亮图标
```

### 线 B：引导 → 红点（弱耦合）
```text
引导开始 → 红点.SetCount("入口.引导", 1)   // 主界面"引导"入口亮红点
引导结束 → 红点.SetCount("入口.引导", 0)   // 灭
```
引导**不拥有**红点逻辑，只是"我有引导要做"的表达；删掉这两行就解耦。

### 线 C：引导靠"事件"推进
```text
玩家完成"释放技能" → 业务发事件 "skill.cast"
                   → 引导监听完成条件命中 → 进入下一步
```
引导只认事件名，不关心谁发的——这是解耦核心。

### 线 D：服务端权威（联机必看）
```text
登录：客户端 → C2S_GetAllRedPoint → 服务端查库下发 S2C_RedPointData
变动：服务端业务变动 → S2C_PushRedPointChange → 客户端更新叶子 → 递归刷父级
点击：本地立即隐藏 → C2S_MarkRedPointRead 异步上报 → 服务端改库（多端推送）
重连：重拉全量，服务端数据校准本地偏差
```

## 3. 解耦的通用手段
- **事件总线（观察者）**：规则层经总线广播"发生了什么"，UI 只决定怎么显示，不反向改规则。
- **简单接口**：业务系统 → 红点/引导，调公开简单接口（`SetCount` / `StartTutorial`）。
- **服务端权威 + 客户端优先**：红点/引导的"真值"在服务端，客户端先响应再同步，兼顾流畅与一致。

## 4. 日志是横切基础设施
- 红点/引导里所有 `log(...)` 自动进日志系统落盘，**业务零侵入**。
- 关键异常用 `logException` 抓栈；崩溃由全局钩子兜底。
- 日志不依赖红点/引导，三者靠"都写日志"自然汇聚，无硬耦合。

## 5. 通用设计原则小结
| 原则 | 做法 |
|---|---|
| UI 只展示不决策 | 规则层经事件总线通知 UI，UI 不反向改规则 |
| 标识用标签/字符串，不用枚举硬编 | 步骤/节点/事件都用 id/tag，可配置 |
| 高频变化靠脏标记攒批 | 红点刷新、日志落盘都异步批量，防抖防卡 |
| 基础设施横切、业务无感知 | 日志拦截现有框架下游，不替换写日志方式 |
| 服务端权威 + 客户端优先 | 联机游戏红点/引导真值在服务端，本地先响再同步 |
| 状态可持久化 | 引导断点存盘续引；红点已读本地缓存+服务端库 |
