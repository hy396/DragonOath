# 通用学习版 · 红点系统

> 与引擎无关。重点讲"树 + OR 聚合 + 脏标记刷新 + 服务端权威"四件事。

## 1. 它解决什么问题

"哪里有我没处理的事？"——邮件、任务、可升级技能……如果每个功能自己亮红点，父菜单还得手动汇总，容易漏、容易错。红点系统用**一棵树**统一表达：父节点红点 = 子节点有任意一个红点（或子节点数值之和）。

### 核心作用
1. 提醒玩家未处理内容，提升活跃与留存
2. 统一规范所有红点逻辑，易维护易扩展
3. 分离业务逻辑与 UI 视图，适配热更新与迭代
4. 依托服务端权威数据管控，防作弊、全服一致

## 2. 系统组成
1. **节点层级**：多叉树划分父子菜单，子功能红点点亮，上级自动同步
2. **数据中心**：统一存红点数量、已读状态、显示样式
3. **管理器**：状态计算、递归刷新、事件分发、前后端同步
4. **UI 表现层**：挂组件绑红点 ID，只展示不写业务判断
5. **状态持久化**：本地缓存已读 + 服务端库实现跨设备统一

## 3. 运行逻辑（行业通用）
1. 数据来源：红点数量/未读由服务端下发，客户端不自主算业务数据
2. 刷新方式：数据变动**触发事件**刷新，无轮询，性能高
3. 联动规则：递归汇总子节点数量赋值父节点，层级联动
4. 清除逻辑：玩家点击后本地即时隐藏，再异步上报服务端同步已读

### 数据结构与遍历
- 结构：多叉树（功能层级树）
- 遍历：自下而上递归，完成父子同步
- 辅助：字典存节点映射、列表存子节点集合

### 核心设计原则（服务端权威）
1. 数据权威在服务端：数量/已读最终以服务端为准
2. **客户端优先响应**：点击先本地隐藏，再同步服务端，保证流畅
3. 层级联动客户端自算，服务端只下发底层业务数值

## 4. 核心数据结构（伪代码）
```text
Node {
  id        : string
  parent    : string        // 父节点 id，空=根
  children  : Node[]
  count     : int           // 待处理数量（0=不亮，>0=亮且可带数字）
  isRead    : bool          // 已读状态（本地缓存用）
  source    : ()=>int       // 可选：拉模式来源，返回该节点数量
  dirty     : bool          // 是否待重算
}

RedDotSystem {
  nodes : Map<id, Node>
  dirty : Set<id>           // 待刷新集合
}
```
> 真实项目里红点常带"数字"（如未读 3 封），所以 `count` 比纯布尔更通用；UI 按 `count>0` 决定显隐，按 `count` 决定显示数字还是小圆点。

## 5. 两种驱动方式
| 方式 | 含义 | 例子 |
|---|---|---|
| **推 Push** | 业务直接说"这个亮/灭（或给数量）" | `SetCount("mail", 3)` |
| **拉 Pull** | 业务给"判断函数"，系统定时问它 | `Bind("skill.upgradable", ()=> hasSkillPoint ? 1 : 0)` |

两种都只做一件事：`dirty.add(id)`，不直接重算。

## 6. 脏标记 + 批量刷新（防抖关键）
```text
markDirty(id): dirty.add(id)              // 只标记

每 0.2 秒（定时器）flush():
  for id in dirty:
      if id.source 存在: id.count = id.source()   // 重新拉取
  for id in dirty:
      recomputeAncestors(id)             // 向上聚合
  for 每个状态变化的节点: notifyUI(id, count)
  dirty.clear()

recomputeAncestors(id):
  cur = id
  while cur 有父 p:
      // 数值型：父.count = Σ 子.count ；布尔型：父 = 任一子
      p.count = sum(子.count)  (或 p.count = 任一子>0 ? 1 : 0)
      cur = p
```
- **聚合语义**：父节点完全由子节点派生，业务只管叶子。
- **防抖**：状态没变就不通知 UI；同值 `SetCount` 直接忽略。

## 7. 前后端交互协议（行业通用，轻量）
### 前端 → 后端
- `C2S_GetAllRedPoint`：登录拉全量（参数 userId）
- `C2S_MarkRedPointRead`：上报已读（参数 redId）

### 后端 → 前端
- `S2C_RedPointData`：登录返回全量
```json
{ "redCount": { "1001": 3, "1002": 2, "1003": 1, "1004": 5 },
  "readList": [1001, 1003] }
```
- `S2C_PushRedPointChange`：实时推送单点变更（参数 redId, count）

## 8. 完整业务流程
1. **登录初始化**：拉全量 → 覆盖本地 → 自下而上递归算父级 → 首次展示
2. **运行中实时更新**：后端业务变动 → 推送单点 → 客户端更新叶子 → 递归刷父级 → UI 刷新
3. **点击清除**：本地立即隐藏 → 异步上报 → 后端改库持久 → 多端在线则推送其他端同步清除
4. **断线重连校准**：重拉全量，用服务端最新数据校准本地，消除偏差

## 9. 通知 UI（观察者模式）
```text
onNodeChanged(id, count):
    广播事件 "reddot.changed" {id, count}
    UI 监听后只更新对应那个红点图标
```
UI 不碰红点逻辑，只听通知刷新图标。

## 10. 常见坑
- **成环**：A 是 B 父、B 又是 A 父 → 重算死循环。注册时检测并拒绝。
- **同值重复通知**：`count` 没变就不广播。
- **高频源抖**：靠 `dirty` 攒批天然解决。
- **权威偏差**：客户端本地隐藏后上报失败，需重连校准（见流程 4）。

## 11. 学习要点小结
- 红点系统**只提供机制**（树/刷新/通知），**不含业务数据**。
- 服务端权威 + 客户端优先响应，是联机游戏标准做法。
- 脏标记攒批 = 高性能；树结构 = 多层菜单零手工维护。
