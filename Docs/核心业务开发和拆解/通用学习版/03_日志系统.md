# 通用学习版 · 日志系统

> 与引擎无关。重点讲"分级、异步落盘、崩溃抓栈、结构化"四件事。
> 类比：游戏=汽车，日志=行车记录仪+故障黑匣子。正常行驶记轨迹，出事立刻记现场原因。

## 1. 它解决什么问题

`print("error")` 只能看一眼就丢。正式项目需要：按重要程度筛选、写进文件留痕、崩溃时还能拿到调用栈定位问题。日志系统是**所有其他系统的横切基础设施**。

### 原生 Debug.Log 的缺陷
1. 临时打印，打包后基本作废，杂乱无章
2. 无分级，全部打印，性能损耗大
3. 无结构化信息，只有纯文本
4. 崩溃捕获有限，只能捕获部分报错
5. 无本地持久化、无自动删除
6. 无法线上上传报错
7. 排查效率极低

### 企业日志系统对比优势
| 对比项 | 原生 Debug.Log | 企业级日志系统 |
|---|---|---|
| 真机查看 | 打包后基本看不到 | 本地存文件 + 后台上报 |
| 性能 | 主线程直接打印，多卡锁 | 全异步队列，0 卡主线程 |
| 日志分级 | 无区分，全打印 | 6 级分级，自由开关 |
| 结构化 | 只有文本 | 时间/模块/设备/版本/玩家ID/堆栈 |
| 崩溃捕获 | 仅捕获部分 | 全局异常 + 崩溃全捕获 |
| 本地存储 | 无限膨胀 | 自动分割、自动删除旧日志 |
| 线上上报 | 无法上传 | 错误日志自动上传 |
| 排查效率 | 极低 | 一键检索定位 |

## 2. 日志分级（Severity）
从重要到啰嗦，常见 6~7 级：
```text
Fatal  > Error > Warning > Info > Debug > Verbose > Trace
 崩溃    出错    警告     信息   调试    冗长     追踪
```
- 每条日志带"级别"，运行时设一个"最低落盘级别"，低于它的不写文件（减噪音）。
- 例：线上设 `Warning`，排查临时调到 `Debug`。

## 3. 结构化内容（一条日志该有什么）
`[时间][级别][模块/通道][玩家ID][设备][版本] 消息 + 堆栈`
- 线上可按玩家 ID 检索其全部行为日志（定位机型兼容、协议错误）。
- 崩溃时自动附：最后执行代码、堆栈、设备软硬件、网络/内存/场景状态。

## 4. 三个组成部件
| 部件 | 职责 |
|---|---|
| **生产者** | 业务调用 `log(level, msg)` |
| **分发器 Sink/Appender** | 按级别过滤，决定写文件/控制台/上报 |
| **存储器** | 落盘，支持滚动、缓冲 |

## 5. 异步落盘（防卡线程关键）
日志可能从**任何线程**产生，不能在产生的瞬间直接写文件。用**生产者-消费者队列**：
```text
log(msg):
    加锁: queue.push(格式化后的行)     // 极快，只入队
    同时: ringBuffer.push(行)         // 内存环形缓冲，供调试面板读

后台定时器每帧/每秒 flush():
    加锁: batch = 取出 queue 全部
    若文件太大 → 滚动（归档旧、开新）
    把 batch 追加写入文件
```
- **环形缓冲**：只留最近 N 条在内存，调试 UI 零 IO 读。
- **滚动 + 自动清理**：单文件超限换新，过期文件自动删，不占存储。
- **线程安全**：多线程打印不崩。
- **退出落盘**：游戏退出时把队列剩余全部写盘，日志不丢。

## 6. 崩溃 / 异常堆栈捕获（黑匣子）
两类：
**(1) 业务主动捕获**（已知可能出错处）：
```text
try { 危险操作 }
catch e: logException("模块名", e.message, stackTrace())
```
`stackTrace()` 抓当前调用栈（回溯调用帧 + 符号化函数名/文件/行号）。

**(2) 引擎级崩溃**（没人 catch）：注册"全局错误钩子"，崩溃时：
```text
onFatalError():
    抓栈 → 入队 → 立刻同步写文件（尽量在进程死前保住现场）
```
无日志，闪退问题 90% 解决不了。

## 7. 上报与线上管控
- **仅上传错误日志**（Fatal/Error），省用户流量。
- **线上动态管控级别**：正式服只开 Info/Warn/Error；灰度服临时开 Debug，不用重新打包。
- **结构化 JSON 日志**，后端直接解析接入。

## 8. 行业用途全景（为什么值得做）
1. **开发阶段**：模块分级、屏蔽冗余、统一格式
2. **测试阶段**：日志反推 BUG 定位代码、还原操作路径、统计隐性 BUG
3. **线上正式服**：按玩家 ID 检索、看设备/版本/场景、精确定位
4. **崩溃溯源**：最后代码 + 堆栈 + 设备 + 场景状态
5. **动态管控级别**：不重新打包调开关
6. **数据分析/埋点**：登录/充值/关卡通过率/卡关场景
7. **合规风控**：追溯外挂、作弊、非法发货，客服有据可查
8. **性能监控**：资源加载耗时、帧异常、网络延迟、内存峰值

## 9. 不重复造轮子
多数引擎已有日志框架（Unity 的 `Debug`/自定义 Logger、Unreal 的 `UE_LOG` + `OutputDevice`）。本系统应**挂在现有框架下游**：拦截它的输出，加"持久化 + 分级过滤 + 堆栈捕获"，不替换业务写日志方式 → 业务零改动。

## 10. 学习要点小结
- 分级：该看的看得到，不该看的别吵。
- 队列 + 后台落盘：不卡主线程、线程安全。
- 崩溃钩子 + 堆栈：线上能破案的最后保险。
- 横切所有系统，但业务方无感知（只管写日志）。
