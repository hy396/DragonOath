# 一、新手引导系统
## 1.1 概念前言
Unity游戏开发中，新手引导是一套用于帮助玩家快速熟悉游戏规则、界面和操作方式的系统。
它通过一系列预设的步骤，逐步引导玩家完成特定的游戏行为，是提升玩家体验、降低流失率的关键功能。

### 市面上三种实现方案
1. 公司中高级开发封装好的闭源新手引导框架
2. 第三方插件工具
3. 项目自研手写框架

## 1.2 企业级开发完整分类
### 两大引导大类
1. **流程式引导（主线强制引导）**
强制步骤、跟随点击、强制解锁功能，卡死流程，必须完成才能推进游戏。
2. **气泡弹窗弱引导**
仅弹窗提示，不卡死主线流程，多用于新系统解锁、活动提示。

### 细分引导类型
- UI点击引导（最常用）
- 3D场景角色引导
- 剧情对话引导
- 玩法规则弹窗引导
- 红点+弹窗组合引导

## 1.3 新手引导完整体系架构
### 1. 整体架构
#### 客户端核心职责
1. 渲染遮罩、高亮、箭头、提示文字
2. 监听引导步骤推进：下一步/跳过/重新触发
3. 监听目标控件：加载、显隐、挂载销毁
4. 步骤条件判断：参数、状态、配置表校验
5. 流程分屏、多段动画、多渠道
6. 引导过程全局事件监听、跳转拦截

#### 服务端（后端）核心三种模式
- **模式A：纯客户端离线引导（小型手游、休闲游戏）**
  1. 后端完全不参与
  2. 引导步骤、进度全部本地缓存
  3. 适合：小游戏、超休闲、微型小游戏、轻度手游
- **模式B：服务端下发引导配置+同步进度（中大型手游通用）**
  1. 配置下发：引导步骤、文案、解锁条件、跳转界面ID，后端配置下发给客户端
  2. 进度同步机制：把每一步完成情况同步玩家数据库
  3. 典型应用：推图主线、任务引导、分阶段开放开启
  4. 进度存档：服务器存、本地缓存，断线重连读取同步
- **模式C：强服务驱动引导（重度MMO、RPG）**
  1. 按服务端指令：战力、副本进度、付费状态动态下发引导分支
  2. 控制动态解锁：双端控制解锁条件，客户端渲染不控制引导
  3. 限制、推送通知：关键节点触发强制校验

### 2. 前后端详细分工（行业标准）
#### 客户端做什么
1. 引导遮罩Shader、镂空高亮、箭头动画
2. 步骤逻辑驱动、状态机流转
3. 读取3D/2D目标、屏幕坐标转换
4. 各类弹窗监听：点击、打开弹窗、关闭弹窗
5. 场景加载/切换事件监听
6. 引导标记、重置开关、重置进度
7. 多语言文案渲染、排版适配
8. 维护引导配置表（Excel配置表）：步骤ID、触发条件、关联UI、弹窗、文案、动画、前置条件
9. 存储玩家引导进度数据（已完成步骤、是否永久跳过）
10. 接收新配置更新后刷新引导开关（等级、关卡、开服天数）
11. 灰度、分服、分渠道差异化引导开关
12. 引导行为埋点日志：进入引导、完成步骤、跳过、关闭流失
13. 关键流程校验拦截器，玩家操作拦截强制等待引导
14. 缓存引导进度，每一步文案、箭头位置
15. 双端触发条件、校验参数组件
16. 配置表读取、配置进度卡点
17. 制作引导预览面板、分页面渲染

#### 后端做什么
1. 客户端初始化 → 拉取全量新手引导基础配置+引导进度
2. 缓存数据返回：已完成引导步骤、未解锁引导、开服天数
3. 客户端GuideManager状态机+服务端进度同步
4. 配置触发条件：达成进度、定位目标UI/3D物体
5. 玩家完成当前步骤 → 同步进度(点击/战斗/升级)
6. 客户端上报步骤完成状态云端
7. 后端入库存档，返回下一步指令
8. 全服进度统一永久持久，不跨进度

### 3. 引导条件类型（前后端都要支持）
#### 客户端本地条件
- 点击指定按钮
- 打开指定弹窗
- 关卡通关
- 完成、倍数、双倍倍数
- 剧情对话点下一步
- 玩家达到指定等级
- 通关关卡/副本
- 获得指定道具/资源
- 开服第N天
- 完成新手任务
- 首次充值、首次交手

#### 服务端下发条件
- 引导分支逻辑：不同玩家走不同引导（新手/回流/老玩家）
- 引导优先级：新引导 > 未完成红点 > 红点提示、弹窗排队
- 防冲突机制：玩家同时多引导，自动过滤低优先级提示
- 防重复引导：结束历史、不重复触发流程
- 埋点数据分析：每一步转化率、流失点、策划迭代优化
- 端游/端服校验配置：不相等卡死，策划在编辑器直接修改，改多端

### 4. 总结规范
- 客户端：负责表现、渲染、遮罩、交互、本地逻辑
- 服务端：负责配置、存档、条件判定、灰度、数据同步、埋点参与
- 小型/轻度游戏：可以纯客户端做，后端零参与
- 中重度手游：必须前后端配合，配置由服务端调控，进度云端保存

## 1.4 配套测试需求
1. 快速调试切换引导步骤
2. 分段测试：只看当前，屏蔽无用测试信息
3. 自动屏蔽代码弹窗，不用手动遮挡弹窗
4. 多人协作开发，统一日志格式，所有人看懂
5. 测试反馈BUG，直接查看日志定位代码位置
6. 记录操作流程+场景+时间，还原玩家操作路径
7. 统计高频警告、隐性BUG（不闪退但逻辑异常）

# 二、红点系统
## 2.1 红点系统基础概念
Unity游戏客户端红点系统：一套统一管理游戏功能提示标记的业务组件，基于逻辑多叉树搭建层级关系，采用数据驱动+事件驱动，自动控制界面小红点、数字提示的显示与隐藏，用来提醒玩家有未读消息、未领奖励、待处理玩法等内容。

### 核心作用
1. 直观提醒玩家未处理游戏内容，提升活跃与留存
2. 统一规范所有红点逻辑，易维护、易扩展
3. 分离业务逻辑与UI视图，适配热更新与项目迭代
4. 依托后端权威数据管控，防止作弊，保证全服状态一致

### 系统组成
1. **节点层级**：多叉树划分父子菜单，子功能红点点亮，上级菜单自动同步点亮
2. **数据中心**：统一存放红点数量、已读状态、显示样式
3. **管理器**：负责状态计算、递归刷新、事件分发、前后端数据同步
4. **UI表现层**：挂载组件绑定红点ID，只做界面展示，不写业务判断
5. **状态持久化**：本地缓存已读记录，搭配服务端数据库实现跨设备统一

### 运行逻辑
1. 数据来源：红点数量、待处理条数由服务端线下下发，客户端不自主计算业务数据
2. 刷新方式：数据变动触发事件刷新，无轮询，性能高
3. 联动规则：递归汇总子节点所有红点数量，赋值给父节点，实现层级联动
4. 清除逻辑：玩家点击后本地即时隐藏红点，再异步上报服务端同步已读状态

### 数据结构与遍历算法
- 结构：多叉树（功能层级树）
- 遍历算法：自下而上递归遍历，完成父子节点状态同步
- 辅助结构：字典存节点映射、列表存子节点集合

### 核心设计原则（服务端权威）
1. 数据权威在服务端：红点数量、已读状态最终以服务端为准
2. 客户端优先响应：点击红点先本地隐藏，再同步同步服务端，保证操作流畅
3. 层级联动客户端自行计算，服务端只下发底层业务数值

## 2.2 前后端交互协议
### 前端 → 后端（客户端发服务端）
1. `C2S_GetAllRedPoint`：请求全量红点数据
   参数：userId（玩家ID）
2. `C2S_MarkRedPointRead`：上报红点已读
   参数：redId（红点ID，枚举对应数字）

### 后端 → 前端（服务端发客户端）
1. `S2C_RedPointData`：登录返回全量红点数据
```json
{
  redCount: {
    1001: 3, //部件未读
    1002: 2, //任务可领
    1003: 1, //活动红点
    1004: 5  //好友申请
  },
  readList: [1001, 1003] //已读红点列表
}
```
2. `S2C_PushRedPointChange`：服务端主动推送红点实时变化
参数：redId（变更红点ID）、count（最新数量）

## 2.3 完整业务流程
### 1. 玩家登录初始化阶段
1. 客户端登录成功，主动发送请求协议，向服务端拉取全游戏红点数据
2. 服务端查询玩家数据表，统计各叶子功能未读/未处理数量、玩家已有查看记录
3. 服务端组装数据下发给客户端
4. 客户端接收数据，覆盖本地旧状态，初始化所有叶子红点数值
5. 客户端通过多叉树自下而上递归，自动计算刷新所有父菜单红点
6. 游戏界面全部红点完成首次展示

### 2. 游戏运行中实时更新流程
1. 服务端产生业务变动（新邮件、好友申请、活动奖励、可领任务）
2. 服务端主动推送协议，单点下发对应叶子红点最新数量
3. 客户端收到数据，直接更新对应叶子节点红点值
4. 自动向上递归刷新各级父节点红点状态
5. 触发界面刷新事件，UI自动更新红点/数字展示

### 3. 玩家点击清除红点流程
1. 玩家点击带红点的按钮/打开对应功能界面
2. 客户端立即执行：本地标记已读、清空红点数值，界面红点瞬间消失
3. 客户端异步发送上报请求，告知服务端该红点已查看
4. 服务端接收上报请求，修改数据库内玩家红点已读状态并持久化存储
5. 服务端返回操作成功回执
6. 多设备同时登录时，服务端向其他在线客户端推送已读状态，全端同步清除红点

### 4. 断线重连校准流程
1. 客户端断线重连成功
2. 重新发送拉取全量红点数据请求
3. 用服务端最新数据校准客户端本地所有红点状态，消除数据偏差

## 2.4 客户端&后端职责拆分
### 客户端职责
1. 搭建多叉树层级结构，实现父子红点递归联动
2. 接收服务端数据，驱动红点状态变化
3. 监听事件，完成UI红点显隐、数字展示
4. 本地缓存已读状态，优先做界面反馈
5. 主动上报已读行为，完成双向状态对齐

### 后端职责
1. 维护玩家红点数据表，存储已读记录
2. 统计各类业务待处理数量
3. 登录批量下发数据，实时推送变动
4. 接收客户端已读请求，更新数据库
5. 实现多端登录状态同步

## 2.5 客户端代码分层结构
### 核心脚本文件
1. `RedPointId.cs`：红点唯一枚举定义，全局所有红点唯一ID，前后端约定统一
2. `RedPointTreeConfig.cs`：红点树形层级配置，定义父子节点关系
3. `RedPointData.cs`：红点本地数据模型，存储数量、已读状态
4. `RedPointProtoData.cs`：网络协议数据结构，和后端交互的数据载体
5. `RedPointManager.cs`：红点核心管理器（包含前后端交互、树递归计算）
6. `UIRedPointItem.cs`：UI红点挂载组件，挂在按钮上自动渲染红点
7. `RedPointGameLogic.cs`：业务调用测试脚本，模拟业务触发红点

### 技术实现亮点
1. **结构**：多叉树管理菜单层级，递归遍历实现父子联动
2. **刷新**：观察者事件驱动，无轮询高性能
3. **交互**：数据权威在服务端，客户端优先界面响应再同步服务端
4. **存储**：本地PlayerPrefs缓存已读，服务端数据库持久化
5. **协议**：只传递红点ID、数量、已读集合轻量数据，流量开销极小

### 交互流程总览
1. 玩家登录 → 请求全量红点数据 → 服务下发全量数据 → 客户端递归刷新所有父菜单红点
2. 服务端业务变动 → 推送单红点更新 → 客户端实时更新红点并联动上级菜单
3. 玩家点击消除红点 → 本地立刻隐藏红点 → 异步上报已读状态给服务端
4. 断线重连 → 重新拉取全量数据，校准本地所有红点状态

# 三、Unity企业级日志系统
## 3.1 日志系统基础定义
日志系统 = 游戏/软件专属「行为记录本+故障黑匣子」，专门用来自动记录程序运行全过程，方便查错、排查问题、追踪流程。

简单类比：游戏=一辆汽车，日志系统=汽车行车记录仪+故障检测仪；正常行驶记录轨迹，出车祸立刻记录现场原因。

### 日志系统核心作用
1. 定位代码报错具体执行步骤
2. 玩家手机闪退，不用真机复现也能定位BUG
3. 区分网络、UI、战斗、登录等模块问题
4. 线上关闭多余调试日志，提升游戏性能
5. 记录玩家操作流程，还原BUG场景

### 原生Debug.Log的缺陷
1. 临时打印，仅编辑器可见，打包后基本作废，杂乱无章
2. 无日志分级，全部打印，性能损耗大
3. 无结构化信息，只有纯文本
4. 崩溃捕获有限，只能捕获部分报错
5. 无本地持久化，无自动删除
6. 无法线上上传报错
7. 排查效率极低

### 企业日志系统对比优势
|对比项|原生Debug.Log|企业级日志系统|
| ---- | ---- | ---- |
|真机查看|打包后基本看不到|本地存文件+后台上报|
|性能|主线程直接打印，多卡锁|全异步队列，0卡主线程|
|日志分级|无区分，全部打印|6级分级，自由开关|
|结构化信息|只有文本|时间/模块/设备/版本/玩家ID/堆栈|
|崩溃捕获|仅捕获部分报错|全局异常+Unity崩溃全捕获|
|本地存储|无管理，无限膨胀|自动分割、自动删除旧日志|
|线上上报|无法上传|错误日志自动上传后台|
|排查效率|极低|一键检索定位BUG|

## 3.2 日志系统核心组成
1. **日志分级**：普通信息、警告、错误、致命崩溃，分开区分
2. **结构化内容**：时间、模块、操作行为、报错堆栈、设备信息、玩家ID
3. **写入方式**：全异步写入，不占用游戏主线程，不卡顿
4. **存储位置**：本地手机文件持久保存日志文件
5. **上报功能**：严重错误自动发给后端服务器
6. **自动管理**：日志太多自动拆分、自动删除旧日志，不占用存储空间

## 3.3 不用企业日志系统的弊端
1. 仅依赖Debug.Log，打包真机看不到日志，玩家手机出问题完全不知道原因
2. 报错无堆栈、无场景、设备信息，线上BUG无法复现、无法定位
3. 所有日志杂乱无章，分不清哪个模块、哪个玩家出的问题
4. 大量调试日志塞满内存/本地存储，占用性能、拖慢游戏
5. 闪退、卡死、黑屏，根本不知道在哪一行代码

## 3.4 游戏行业日志系统核心用途
### 1. 开发阶段：快速调试排错
区分模块日志；自动屏蔽冗余调试信息；多人协作统一日志格式。

### 2. 测试阶段：自动化定位问题
1. 日志反推BUG，直接定位代码位置
2. 记录操作流程+场景+时间，还原玩家操作路径
3. 统计高频隐性BUG（不闪退但逻辑异常）

### 3. 线上正式服（最重要）
1. 抓取玩家UID/角色ID，直接检索玩家全部行为日志
2. 查看当时设备型号、系统版本、游戏版本、所在场景
3. 精确定位：代码BUG、配置错误、协议错误、机型兼容问题

### 4. 崩溃闪退溯源（黑匣子作用）
游戏闪退瞬间自动记录：
1. 最后执行哪段代码
2. 堆栈报错代码
3. 设备软硬件信息
4. 当前网络、内存、场景状态
无日志，闪退问题90%解决不了。

### 5. 线上动态管控日志级别
正式服只开启Info/Warn/Error关键错误日志，提升性能、减少流量；灰度服/私服临时开启Debug日志，排查疑难BUG；不用重新打包，直接配置控制日志开关。

### 6. 数据分析&行为埋点
1. 玩家登录、进服、充值、创角率
2. 关卡通过率、付费点击行为
3. 卡关集中场景、弱网异常情况
4. 辅助策划优化游戏体验、调整数值。

### 7. 合规&运营风控
1. 日志留存可追溯异常账号、外挂行为、恶意刷资源
2. 排查协议篡改、作弊请求、非法发货
3. 客服处理玩家投诉，有据可查。

### 8. 性能监控与优化
日志记录：
1. 资源加载耗时
2. 帧同步异常
3. 网络延迟波动
4. 内存峰值
用来针对性做游戏性能优化、搭配机型适配。

## 3.5 日志系统客户端代码分层结构

9. `LogTest.cs`：测试脚本，挂载即用快速调试

4. 本地滚动日志（自动切分、自动清理过期文件）

6. 仅上传错误日志，节省用户流量
7. 线程安全，多线程打印不崩溃
8. 游戏退出自动落盘，日志永不丢失
9. 自带设备、版本、场景、玩家信息自动附加

# 四、整体架构总览（三大系统统一总结）
## 4.1 三大系统通用企业级标准
1. **前后端分离架构**
   - 客户端：负责UI渲染、表现、本地缓存、事件驱动、界面交互
   - 服务端：数据权威、配置下发、进度持久化、多端同步、风控埋点
2. **数据驱动+事件驱动**
   无轮询刷新，数据变更触发界面自动更新，低性能消耗
3. **本地缓存+云端持久双保障**
   本地优先响应提升操作流畅度，后端做最终数据校准防作弊、跨设备同步
4. **配置表解耦业务**
   引导步骤、红点ID、日志分级全部表格配置，支持热更新、策划自由修改不用改代码
5. **完善埋点与日志配套**
   所有系统行为全埋点日志，线上问题可追溯、玩家行为可数据分析
6. **模块化分层代码**
   接口隔离、业务逻辑与表现层分离，易维护、可复用、支持迭代扩展

## 4.2 各系统技术选型总结
|系统|核心数据结构|同步方案|性能优化点|
| ---- | ---- | ---- | ---- |
|新手引导|状态机+步骤配置表|中大型游戏前后端同步；轻度纯本地|步骤预加载、引导遮挡屏蔽无关UI、异步配置读取|

# 一、新手引导系统
## 1.1 概念前言
Unity游戏开发中，新手引导是一套用于帮助玩家快速熟悉游戏规则、界面和操作方式的系统。
它通过一系列预设的步骤，逐步引导玩家完成特定的游戏行为，是提升玩家体验、降低流失率的关键功能。

### 市面上三种实现方案
1. 公司中高级开发封装好的闭源新手引导框架
2. 第三方插件工具
3. 项目自研手写框架

## 1.2 企业级开发完整分类
### 两大引导大类
1. **流程式引导（主线强制引导）**
强制步骤、跟随点击、强制解锁功能，卡死流程，必须完成才能推进游戏。
2. **气泡弹窗弱引导**
仅弹窗提示，不卡死主线流程，多用于新系统解锁、活动提示。

### 细分引导类型
- UI点击引导（最常用）
- 3D场景角色引导
- 剧情对话引导
- 玩法规则弹窗引导
- 红点+弹窗组合引导

## 1.3 新手引导完整体系架构
### 1. 整体架构
#### 客户端核心职责
1. 渲染遮罩、高亮、箭头、提示文字
2. 监听引导步骤推进：下一步/跳过/重新触发
3. 监听目标控件：加载、显隐、挂载销毁
4. 步骤条件判断：参数、状态、配置表校验
5. 流程分屏、多段动画、多渠道
6. 引导过程全局事件监听、跳转拦截

#### 服务端（后端）核心三种模式
- **模式A：纯客户端离线引导（小型手游、休闲游戏）**
  1. 后端完全不参与
  2. 引导步骤、进度全部本地缓存
  3. 适合：小游戏、超休闲、微型小游戏、轻度手游
- **模式B：服务端下发引导配置+同步进度（中大型手游通用）**
  1. 配置下发：引导步骤、文案、解锁条件、跳转界面ID，后端配置下发给客户端
  2. 进度同步机制：把每一步完成情况同步玩家数据库
  3. 典型应用：推图主线、任务引导、分阶段开放开启
  4. 进度存档：服务器存、本地缓存，断线重连读取同步
- **模式C：强服务驱动引导（重度MMO、RPG）**
  1. 按服务端指令：战力、副本进度、付费状态动态下发引导分支
  2. 控制动态解锁：双端控制解锁条件，客户端渲染不控制引导
  3. 限制、推送通知：关键节点触发强制校验

### 2. 前后端详细分工（行业标准）
#### 客户端做什么
1. 引导遮罩Shader、镂空高亮、箭头动画
2. 步骤逻辑驱动、状态机流转
3. 读取3D/2D目标、屏幕坐标转换
4. 各类弹窗监听：点击、打开弹窗、关闭弹窗
5. 场景加载/切换事件监听
6. 引导标记、重置开关、重置进度
7. 多语言文案渲染、排版适配
8. 维护引导配置表（Excel配置表）：步骤ID、触发条件、关联UI、弹窗、文案、动画、前置条件
9. 存储玩家引导进度数据（已完成步骤、是否永久跳过）
10. 接收新配置更新后刷新引导开关（等级、关卡、开服天数）
11. 灰度、分服、分渠道差异化引导开关
12. 引导行为埋点日志：进入引导、完成步骤、跳过、关闭流失
13. 关键流程校验拦截器，玩家操作拦截强制等待引导
14. 缓存引导进度，每一步文案、箭头位置
15. 双端触发条件、校验参数组件
16. 配置表读取、配置进度卡点
17. 制作引导预览面板、分页面渲染

#### 后端做什么
1. 客户端初始化 → 拉取全量新手引导基础配置+引导进度
2. 缓存数据返回：已完成引导步骤、未解锁引导、开服天数
3. 客户端GuideManager状态机+服务端进度同步
4. 配置触发条件：达成进度、定位目标UI/3D物体
5. 玩家完成当前步骤 → 同步进度(点击/战斗/升级)
6. 客户端上报步骤完成状态云端
7. 后端入库存档，返回下一步指令
8. 全服进度统一永久持久，不跨进度

### 3. 引导条件类型（前后端都要支持）
#### 客户端本地条件
- 点击指定按钮
- 打开指定弹窗
- 关卡通关
- 完成、倍数、双倍倍数
- 剧情对话点下一步
- 玩家达到指定等级
- 通关关卡/副本
- 获得指定道具/资源
- 开服第N天
- 完成新手任务
- 首次充值、首次交手

#### 服务端下发条件
- 引导分支逻辑：不同玩家走不同引导（新手/回流/老玩家）
- 引导优先级：新引导 > 未完成红点 > 红点提示、弹窗排队
- 防冲突机制：玩家同时多引导，自动过滤低优先级提示
- 防重复引导：结束历史、不重复触发流程
- 埋点数据分析：每一步转化率、流失点、策划迭代优化
- 端游/端服校验配置：不相等卡死，策划在编辑器直接修改，改多端

### 4. 总结规范
- 客户端：负责表现、渲染、遮罩、交互、本地逻辑
- 服务端：负责配置、存档、条件判定、灰度、数据同步、埋点参与
- 小型/轻度游戏：可以纯客户端做，后端零参与
- 中重度手游：必须前后端配合，配置由服务端调控，进度云端保存

## 1.4 配套测试需求
1. 快速调试切换引导步骤
2. 分段测试：只看当前，屏蔽无用测试信息
3. 自动屏蔽代码弹窗，不用手动遮挡弹窗
4. 多人协作开发，统一日志格式，所有人看懂
5. 测试反馈BUG，直接查看日志定位代码位置
6. 记录操作流程+场景+时间，还原玩家操作路径
7. 统计高频警告、隐性BUG（不闪退但逻辑异常）

# 二、红点系统
## 2.1 红点系统基础概念
Unity游戏客户端红点系统：一套统一管理游戏功能提示标记的业务组件，基于逻辑多叉树搭建层级关系，采用数据驱动+事件驱动，自动控制界面小红点、数字提示的显示与隐藏，用来提醒玩家有未读消息、未领奖励、待处理玩法等内容。

### 核心作用
1. 直观提醒玩家未处理游戏内容，提升活跃与留存
2. 统一规范所有红点逻辑，易维护、易扩展
3. 分离业务逻辑与UI视图，适配热更新与项目迭代
4. 依托后端权威数据管控，防止作弊，保证全服状态一致

### 系统组成
1. **节点层级**：多叉树划分父子菜单，子功能红点点亮，上级菜单自动同步点亮
2. **数据中心**：统一存放红点数量、已读状态、显示样式
3. **管理器**：负责状态计算、递归刷新、事件分发、前后端数据同步
4. **UI表现层**：挂载组件绑定红点ID，只做界面展示，不写业务判断
5. **状态持久化**：本地缓存已读记录，搭配服务端数据库实现跨设备统一

### 运行逻辑
1. 数据来源：红点数量、待处理条数由服务端线下下发，客户端不自主计算业务数据
2. 刷新方式：数据变动触发事件刷新，无轮询，性能高
3. 联动规则：递归汇总子节点所有红点数量，赋值给父节点，实现层级联动
4. 清除逻辑：玩家点击后本地即时隐藏红点，再异步上报服务端同步已读状态

### 数据结构与遍历算法
- 结构：多叉树（功能层级树）
- 遍历算法：自下而上递归遍历，完成父子节点状态同步
- 辅助结构：字典存节点映射、列表存子节点集合

### 核心设计原则（服务端权威）
1. 数据权威在服务端：红点数量、已读状态最终以服务端为准
2. 客户端优先响应：点击红点先本地隐藏，再同步同步服务端，保证操作流畅
3. 层级联动客户端自行计算，服务端只下发底层业务数值

## 2.2 前后端交互协议
### 前端 → 后端（客户端发服务端）
1. `C2S_GetAllRedPoint`：请求全量红点数据
   参数：userId（玩家ID）
2. `C2S_MarkRedPointRead`：上报红点已读
   参数：redId（红点ID，枚举对应数字）

### 后端 → 前端（服务端发客户端）
1. `S2C_RedPointData`：登录返回全量红点数据
```json
{
  "redCount": {
    "1001": 3, //部件未读
    "1002": 2, //任务可领
    "1003": 1, //活动红点
    "1004": 5  //好友申请
  },
  "readList": [1001, 1003] //已读红点列表
}
```
2. `S2C_PushRedPointChange`：服务端主动推送红点实时变化
参数：redId（变更红点ID）、count（最新数量）

## 2.3 完整业务流程
### 1. 玩家登录初始化阶段
1. 客户端登录成功，主动发送请求协议，向服务端拉取全游戏红点数据
2. 服务端查询玩家数据表，统计各叶子功能未读/未处理数量、玩家已有查看记录
3. 服务端组装数据下发给客户端
4. 客户端接收数据，覆盖本地旧状态，初始化所有叶子红点数值
5. 客户端通过多叉树自下而上递归，自动计算刷新所有父菜单红点
6. 游戏界面全部红点完成首次展示

### 2. 游戏运行中实时更新流程
1. 服务端产生业务变动（新邮件、好友申请、活动奖励、可领任务）
2. 服务端主动推送协议，单点下发对应叶子红点最新数量
3. 客户端收到数据，直接更新对应叶子节点红点值
4. 自动向上递归刷新各级父节点红点状态
5. 触发界面刷新事件，UI自动更新红点/数字展示

### 3. 玩家点击清除红点流程
1. 玩家点击带红点的按钮/打开对应功能界面
2. 客户端立即执行：本地标记已读、清空红点数值，界面红点瞬间消失
3. 客户端异步发送上报请求，告知服务端该红点已查看
4. 服务端接收上报请求，修改数据库内玩家红点已读状态并持久化存储
5. 服务端返回操作成功回执
6. 多设备同时登录时，服务端向其他在线客户端推送已读状态，全端同步清除红点

### 4. 断线重连校准流程
1. 客户端断线重连成功
2. 重新发送拉取全量红点数据请求
3. 用服务端最新数据校准客户端本地所有红点状态，消除数据偏差

## 2.4 客户端&后端职责拆分
### 客户端职责
1. 搭建多叉树层级结构，实现父子红点递归联动
2. 接收服务端数据，驱动红点状态变化
3. 监听事件，完成UI红点显隐、数字展示
4. 本地缓存已读状态，优先做界面反馈
5. 主动上报已读行为，完成双向状态对齐

### 后端职责
1. 维护玩家红点数据表，存储已读记录
2. 统计各类业务待处理数量
3. 登录批量下发数据，实时推送变动
4. 接收客户端已读请求，更新数据库
5. 实现多端登录状态同步

## 2.5 客户端代码分层结构
### 核心脚本文件
1. `RedPointId.cs`：红点唯一枚举定义，全局所有红点唯一ID，前后端约定统一
2. `RedPointTreeConfig.cs`：红点树形层级配置，定义父子节点关系
3. `RedPointData.cs`：红点本地数据模型，存储数量、已读状态
4. `RedPointProtoData.cs`：网络协议数据结构，和后端交互的数据载体
5. `RedPointManager.cs`：红点核心管理器（包含前后端交互、树递归计算）
6. `UIRedPointItem.cs`：UI红点挂载组件，挂在按钮上自动渲染红点
7. `RedPointGameLogic.cs`：业务调用测试脚本，模拟业务触发红点

### 技术实现亮点
1. **结构**：多叉树管理菜单层级，递归遍历实现父子联动
2. **刷新**：观察者事件驱动，无轮询高性能
3. **交互**：数据权威在服务端，客户端优先界面响应再同步服务端
4. **存储**：本地PlayerPrefs缓存已读，服务端数据库持久化
5. **协议**：只传递红点ID、数量、已读集合轻量数据，流量开销极小

### 交互流程总览
1. 玩家登录 → 请求全量红点数据 → 服务下发全量数据 → 客户端递归刷新所有父菜单红点
2. 服务端业务变动 → 推送单红点更新 → 客户端实时更新红点并联动上级菜单
3. 玩家点击消除红点 → 本地立刻隐藏红点 → 异步上报已读状态给服务端
4. 断线重连 → 重新拉取全量数据，校准本地所有红点状态

# 三、Unity企业级日志系统
## 3.1 日志系统基础定义
日志系统 = 游戏/软件专属「行为记录本+故障黑匣子」，专门用来自动记录程序运行全过程，方便查错、排查问题、追踪流程。

简单类比：游戏=一辆汽车，日志系统=汽车行车记录仪+故障检测仪；正常行驶记录轨迹，出车祸立刻记录现场原因。

### 日志系统核心作用
1. 定位代码报错具体执行步骤
2. 玩家手机闪退，不用真机复现也能定位BUG
3. 区分网络、UI、战斗、登录等模块问题
4. 线上关闭多余调试日志，提升游戏性能
5. 记录玩家操作流程，还原BUG场景

### 原生Debug.Log的缺陷
1. 临时打印，仅编辑器可见，打包后基本作废，杂乱无章
2. 无日志分级，全部打印，性能损耗大
3. 无结构化信息，只有纯文本
4. 崩溃捕获有限，只能捕获部分报错
5. 无本地持久化，无自动删除
6. 无法线上上传报错
7. 排查效率极低

### 企业日志系统对比优势
|对比项|原生Debug.Log|企业级日志系统|
| ---- | ---- | ---- |
|真机查看|打包后基本看不到|本地存文件+后台上报|
|性能|主线程直接打印，多卡锁|全异步队列，0卡主线程|
|日志分级|无区分，全部打印|6级分级，自由开关|
|结构化信息|只有文本|时间/模块/设备/版本/玩家ID/堆栈|
|崩溃捕获|仅捕获部分报错|全局异常+Unity崩溃全捕获|
|本地存储|无管理，无限膨胀|自动分割、自动删除旧日志|
|线上上报|无法上传|错误日志自动上传后台|
|排查效率|极低|一键检索定位BUG|

## 3.2 日志系统核心组成
1. **日志分级**：普通信息、警告、错误、致命崩溃，分开区分
2. **结构化内容**：时间、模块、操作行为、报错堆栈、设备信息、玩家ID
3. **写入方式**：全异步写入，不占用游戏主线程，不卡顿
4. **存储位置**：本地手机文件持久保存日志文件
5. **上报功能**：严重错误自动发给后端服务器
6. **自动管理**：日志太多自动拆分、自动删除旧日志，不占用存储空间

## 3.3 不用企业日志系统的弊端
1. 仅依赖Debug.Log，打包真机看不到日志，玩家手机出问题完全不知道原因
2. 报错无堆栈、无场景、设备信息，线上BUG无法复现、无法定位
3. 所有日志杂乱无章，分不清哪个模块、哪个玩家出的问题
4. 大量调试日志塞满内存/本地存储，占用性能、拖慢游戏
5. 闪退、卡死、黑屏，根本不知道在哪一行代码

## 3.4 游戏行业日志系统核心用途
### 1. 开发阶段：快速调试排错
区分模块日志；自动屏蔽冗余调试信息；多人协作统一日志格式。

### 2. 测试阶段：自动化定位问题
1. 日志反推BUG，直接定位代码位置
2. 记录操作流程+场景+时间，还原玩家操作路径
3. 统计高频隐性BUG（不闪退但逻辑异常）

### 3. 线上正式服（最重要）
1. 抓取玩家UID/角色ID，直接检索玩家全部行为日志
2. 查看当时设备型号、系统版本、游戏版本、所在场景
3. 精确定位：代码BUG、配置错误、协议错误、机型兼容问题

### 4. 崩溃闪退溯源（黑匣子作用）
游戏闪退瞬间自动记录：
1. 最后执行哪段代码
2. 堆栈报错代码
3. 设备软硬件信息
4. 当前网络、内存、场景状态
无日志，闪退问题90%解决不了。

### 5. 线上动态管控日志级别
正式服只开启Info/Warn/Error关键错误日志，提升性能、减少流量；灰度服/私服临时开启Debug日志，排查疑难BUG；不用重新打包，直接配置控制日志开关。

### 6. 数据分析&行为埋点
1. 玩家登录、进服、充值、创角率
2. 关卡通过率、付费点击行为
3. 卡关集中场景、弱网异常情况
4. 辅助策划优化游戏体验、调整数值。

### 7. 合规&运营风控
1. 日志留存可追溯异常账号、外挂行为、恶意刷资源
2. 排查协议篡改、作弊请求、非法发货
3. 客服处理玩家投诉，有据可查。

### 8. 性能监控与优化
日志记录：
1. 资源加载耗时
2. 帧同步异常
3. 网络延迟波动
4. 内存峰值
用来针对性做游戏性能优化、搭配机型适配。

## 3.5 日志系统客户端代码分层结构
### 核心脚本文件
1. `LogLevel.cs`：日志等级枚举（Debug/Info/Warn/Error/Fatal等6级）
2. `LogEntity.cs`：日志结构化实体，存储单条日志完整信息
3. `ILogOutput.cs`：日志输出器接口，统一规范输出行为
4. `ConsoleOutput.cs`：编辑器控制台输出实现
5. `FileOutput.cs`：本地滚动日志文件持久化
6. `LogUploader.cs`：日志上传工具，处理报错日志网络上报
7. `NetworkOutput.cs`：网络上报输出实现
8. `GameLogger.cs`：全局日志核心、唯一调用入口（静态单例）
9. `LogTest.cs`：测试脚本，挂载即用快速调试

### 系统功能列表
1. 全异步，绝不卡顿主线程
2. 分级控制，线上可关闭调试日志
3. 自动捕获Unity报错+全局崩溃
4. 本地滚动日志（自动切分、自动清理过期文件）
5. 结构化JSON日志，后端直接解析接入
6. 仅上传错误日志，节省用户流量
7. 线程安全，多线程打印不崩溃
8. 游戏退出自动落盘，日志永不丢失
9. 自带设备、版本、场景、玩家信息自动附加

# 四、整体架构总览（三大系统统一总结）
## 4.1 三大系统通用企业级标准
1. **前后端分离架构**
   - 客户端：负责UI渲染、表现、本地缓存、事件驱动、界面交互
   - 服务端：数据权威、配置下发、进度持久化、多端同步、风控埋点
2. **数据驱动+事件驱动**
   无轮询刷新，数据变更触发界面自动更新，低性能消耗
3. **本地缓存+云端持久双保障**
   本地优先响应提升操作流畅度，后端做最终数据校准防作弊、跨设备同步
4. **配置表解耦业务**
   引导步骤、红点ID、日志分级全部表格配置，支持热更新、策划自由修改不用改代码
5. **完善埋点与日志配套**
   所有系统行为全埋点日志，线上问题可追溯、玩家行为可数据分析
6. **模块化分层代码**
   接口隔离、业务逻辑与表现层分离，易维护、可复用、支持迭代扩展

## 4.2 各系统技术选型总结
|系统|核心数据结构|同步方案|性能优化点|
| ---- | ---- | ---- | ---- |
|新手引导|状态机+步骤配置表|中大型游戏前后端同步；轻度纯本地|步骤预加载、引导遮挡屏蔽无关UI、异步配置读取|
|红点系统|多叉树层级结构|服务端下发增量数据、客户端递归计算|事件驱动刷新、无轮询、轻量协议|
|日志系统|结构化实体队列|仅错误日志异步上报|全异步队列、本地分片存储、分级开关控制打印量|

## 4.3 项目落地适配建议
1. 小型休闲手游：红点+日志必须自研；新手引导可纯客户端离线实现，后端不参与
2. 中重度RPG/MMO：三大系统全部前后端联动，服务端管控配置与进度，支持多端、灰度、埋点数据分析
3. 上线规范：打包发布关闭Debug级日志，仅保留警告/错误日志，减少性能损耗与流量消耗
4. 扩展性预留：所有ID、步骤、配置全部枚举/表格化，后续新增活动、功能无需大规模修改底层代码
