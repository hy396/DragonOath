# 项目相关版 · 红点系统

> 落地于 DragonOath：树状节点 + 脏标记批量刷新 + `GameplayMessageRouter` 同步 UI。
> 子系统归属 `ULocalPlayerSubsystem`（每玩家本地视图）；**权威真值存 `ADOPlayerState`**（联机 RPG 走服务端权威）。
> UE 专属翻译：朋友资料的"后端" = DragonOath 的 DS/Listen Server 权威逻辑 + `PlayerState` 复制。

## 0. 服务端权威在 UE 里怎么落（关键翻译）
朋友资料的红点协议（`C2S_GetAllRedPoint` / `S2C_RedPointData` / `S2C_PushRedPointChange` / `C2S_MarkRedPointRead`）映射到 UE：
- **权威数据**：每玩家的红点"已读状态 + 底层数量"存 `ADOPlayerState`（`Replicated`），服务端是唯一真值。
- **下发配置**：登录时客户端读 `PlayerState` 已复制的 `FDORedDotState` 即等于 `S2C_RedPointData`（属性复制自带，无需手写协议）。
- **实时推送**：服务端改 `PlayerState` 属性 → `OnRep` 触发 = `S2C_PushRedPointChange`。
- **上报已读**：客户端 `Server RPC`（如 `Server_MarkRedPointRead`） = `C2S_MarkRedPointRead`。
- **层级联动（OR/聚合）**：纯客户端算（服务端只下发叶子底层数值），与资料一致。

## 2. 为什么 LocalPlayerSubsystem + PlayerState 双归属
- 红点反映"这个玩家有哪些未处理的事"，是**每玩家**状态：本地视图用 `ULocalPlayerSubsystem`（联机各客户端算自己的）。
- 已读/数量真值用 `ADOPlayerState`：服务端决定，复制下发，防作弊、多端一致（资料"服务端权威"原则）。
- 与引导同源归属，便于"引导进行中→点亮入口红点"联动（都在 LocalPlayer 上，直接 `Get` 互调）。

## 3. 节点结构（真实）
```cpp
struct FDORedDotNode
{
    FGameplayTag Id;                 // 节点标识（GameplayTag）
    FGameplayTag ParentId;           // 父节点；空=根
    TSet<FGameplayTag> Children;     // 子节点
    int32 Count = 0;                 // 待处理数量（0=不亮，>0=亮，可带数字）
    bool bExplicitActive = false;    // 推模式直接给出的亮灭
    bool bComputedActive = false;    // = bExplicitActive || 任一子节点亮
    TFunction<int32()> Evaluator;    // 拉模式来源（返回数量）
    bool bDirty = false;             // 来源变化后置位，待重算
};

// DORedDotSubsystem 成员：
TMap<FGameplayTag, FDORedDotNode> Nodes;
TSet<FGameplayTag> DirtyNodes;       // 待重算集合
FTimerHandle RefreshTimer;           // 默认 0.2s 批量刷新

// 权威端：存于 ADOPlayerState（Replicated）
USTRUCT()
struct FDORedDotState
{
    GENERATED_BODY()
    UPROPERTY(Replicated) TSet<FGameplayTag> ReadNodes;        // 已读集合（真值）
    UPROPERTY(Replicated) TMap<FGameplayTag, int32> LeafCounts; // 底层业务数量（真值）
};
```

## 4. 真实数据源咬合点（DragonOath）
| 红点节点 | 来源（真实系统） | 触发方式 |
|---|---|---|
| `RedDot.Root.Skill.Upgradable` | 技能可升级 | 监听 `UDOAbilitySystemComponent::OnAbilityLevelChanged` + 未来技能点属性 → `MarkNodeDirty` |
| `RedDot.Root.Tutorial` | 引导进行中 | `DOTutorialSubsystem` 调 `SetNodeActive(true/false)` |
| `RedDot.Root.Mail.*` | 邮件系统（未来） | 邮件 `PlayerState` 复制 OnRep → `SetNodeCount` |
| `RedDot.Root.Quest.Daily` | 日常任务（未来） | 任务状态变化 → `MarkNodeDirty` |

> 红点系统**只提供机制**（树/刷新/广播），具体数据源由业务系统在合适时机 `SetNodeActive` / `MarkNodeDirty`。业务系统不碰 UI。

## 5. 核心逻辑
```text
RegisterNode(Id, ParentId)        // 挂到父 Children；父不存在惰性建占位
BindNodeSource(Id, Eval)          // 设拉模式来源 → MarkNodeDirty(Id)
SetNodeActive(Id, b)              // 推模式显式覆盖 → MarkNodeDirty(Id)
SetNodeCount(Id, N)               // 推模式带数字 → MarkNodeDirty(Id)

MarkNodeDirty(Id): DirtyNodes.Add(Id)        // 仅标记，不立即重算（攒批防抖）

RefreshTimer → FlushDirty():
  for Id in DirtyNodes:
      if Evaluator 有效: Count = Evaluator()
  for Id in DirtyNodes: RecomputeAncestors(Id)
  RecomputeAncestors: 自底向上 父.Count = Σ子.Count（或 父.bActive = 任一子）
                       若变化 → BroadcastMessage(Message.UI.RedDot.Changed,{Id,Count})
  DirtyNodes.Reset()
```
- **聚合语义**：父完全由子派生，业务只维护叶子。
- **防抖**：没变化不广播；同值 `Set*` 直接忽略。
- **点击清除**：UI 点红点 → 本地立即 `SetNodeCount(Id,0)`（优先响应）→ `Server RPC` 上报 `PlayerState.ReadNodes.Add(Id)`（异步同步真值）。

## 6. 与消息总线对接
- 变化 → `BroadcastMessage(Message.UI.RedDot.Changed, FDORedDotChangedMessage{NodeId, Count})`。
- UI（UMG/Common UI）`NativeConstruct` 注册 `RegisterListener`，`NativeDestruct` `Unregister`（见 `Docs/04_Local_Message_Bus.md`）。
- 异常（节点成环）→ 经日志系统记录（`DO_LOG_EXCEPTION`）。

## 8. 联机权威校验（实现后验证）
- PIE Listen Server + 1 Client：Client 点红点本地立即灭 → Server `PlayerState.ReadNodes` 更新 → 另一 Client 经 `OnRep` 同步灭（多端一致）。
- 断线重连：重连后 `PlayerState` 复制带最新 `ReadNodes`/`LeafCounts`，本地树校准。
