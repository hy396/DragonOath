# 技能升级联网请求与校验方案

> 参考：Crunch 项目的 `UCAbilitySystemComponent::Server_UpgradeAbilityWithID(ECAbilityInputID)` 方案。
> 范围：本方案只做设计文档与代码接口占位，**不实现 UI**（UI 需要做什么在第七、八节详述）。
> 审批后再进入执行阶段改动代码。

---

## 一、背景与结论

Crunch 的技能升级链路：

```text
客户端 UI
  -> Server_UpgradeAbilityWithID(ECAbilityInputID)   // UFUNCTION(Server, Reliable, WithValidation)
  -> 服务端校验升级点/角色等级
  -> AbilitySpec->Level++ + MarkAbilitySpecDirty
  -> Client_AbilitySpecLevelUpdated(Handle, NewLevel) // 显式 Client RPC 刷 UI
```

**可取之处（本方案采纳）**：
1. **客户端升级请求必须走 Server RPC** —— 升级是服务器权威操作，客户端 UI 不能直接改 `Spec->Level`，需要一条 `Server` RPC 把"升级哪技能、到几级"的请求送到权威端。
2. **服务端校验放在权威端**（`_Validate` + 实现里判升级点/等级/前置），这是防作弊与数据一致性的正确位置。

**不照搬（与本项目规则冲突或冗余）**：
1. **`ECAbilityInputID` 枚举标识技能** —— 违反 `AGENTS.md` 的"不要用枚举做技能/职业标识，使用 `GameplayTag`"。DragonOath 已用 `FGameplayTag AbilityId` 标识技能（见 `FDOAbilityGrant::AbilityId`），**继续使用 Tag 标识，不引入输入枚举**。
2. **手写 `Client_AbilitySpecLevelUpdated` 回包** —— 多余。DragonOath 玩家 ASC 用 `Mixed` 复制模式，`MarkAbilitySpecDirty` 已自动把 `Spec.Level` 复制到客户端。UI 刷新走 `GameplayMessageRouter`（`Message.UI.Skill.*`），不另写 Client RPC。

**一句话**：学 Crunch 的「客户端升级请求 RPC + 服务端校验」这条链路；不学它的 InputID 标识和手写 Client 回包；标识继续用 `AbilityId` Tag，UI 同步复用 GAS 复制 + Message 总线。

## 三、设计原则

1. **服务器权威**：升级只在服务端改 `Spec->Level`；客户端只能通过 `Server` RPC 发起请求。
2. **标识用 Tag**：继续 `FGameplayTag AbilityId`，不引入 `ECAbilityInputID` 类枚举。
3. **不重复造网络**：等级同步依赖 GAS `Mixed` 复制；UI 刷新走 `GameplayMessageRouter`，不写手写 Client RPC。
4. **校验前置化**：所有"能不能升、升到几级、花多少点"的判定在服务端 `_Validate` + `_Implementation` 内完成。
5. **数据驱动**：`MaxLevel`/前置/消耗尽量来自 `AbilitySet`/`SkillTree` 配置，不硬编码。

## 四、服务端接口设计（执行阶段改动）

### 4.1 新增 Server RPC（客户端升级请求入口）

在 `UDOAbilitySystemComponent` 新增：

```cpp
// 客户端请求升级技能。必须经由服务器权威校验后才会真正改 Level。
// 客户端 UI 调用此函数，不要直接调 SetDOAbilityLevel。
UFUNCTION(Server, Reliable, WithValidation)
void Server_RequestAbilityLevel(FGameplayTag AbilityId, int32 NewLevel);
```

- `_Validate`：基础合法性 —— `AbilityId.IsValid()`、`NewLevel > 0`、请求等级不超过该技能 `MaxLevel`（从 `AbilityIdToSpecHandle` 反查授予配置）。
- `_Implementation`：完整权威校验（见 4.3）→ 通过才调 `SetDOAbilityLevel(AbilityId, NewLevel)`；不通过则忽略（可选向客户端回一条失败 Message，见 4.4）。

> 位置说明：玩家 ASC 挂在 `ADOPlayerState` 上（`SetIsReplicated(true)`），`Server` RPC 在 Component 上可正常路由到拥有者连接。若团队更习惯把游戏性 RPC 放 `PlayerController`/`PlayerState`，亦可等价放置在 `ADOPlayerState` 上再转调 ASC，二者皆可，本方案默认放 ASC。

### 4.2 保留并增强 `SetDOAbilityLevel`

现有函数逻辑（0↔1 激活/取消、`MarkAbilitySpecDirty`）**完全保留**，新增：

- 在 `Spec->Level = NewLevel` 成功后，Broadcast 等级变化（见 4.4 委托）。

### 4.3 服务端权威校验清单（放在 `Server_RequestAbilityLevel_Implementation`）

| 校验项 | 说明 | 数据来源 |
|---|---|---|
| 技能已授予 | `AbilityIdToSpecHandle` 含 `AbilityId` | ASC 自身 |
| 等级区间 | `1 <= NewLevel <= Grant.MaxLevel` | `FDOAbilityGrant::MaxLevel` |
| 等级有变化 | `NewLevel != OldLevel`，避免空转 | Spec 当前 Level |
| 技能点足够 | 消耗 = `NewLevel - OldLevel` 点，`AvailablePoints >= 消耗` | 技能点经济（4.5） |
| 角色等级前置 | 若技能要求最低角色等级，`CharacterLevel >= 要求` | `DOCharacter::GetCharacterLevel` |
| 技能树前置 | 依赖其他技能达到某等级 / 职业匹配 | `SkillTreeComponent`（占位，4.6） |

任一不通过则拒绝升级，不扣点、不改 Level。

### 4.4 新增等级变化广播（供 UI / 其他系统订阅）

```cpp
// 技能等级变化时广播（服务端触发，客户端通过复制后的 Spec 已是最新值后监听）。
DECLARE_MULTICAST_DELEGATE_ThreeParams(FOnDOAbilityLevelChanged, FGameplayTag /*AbilityId*/, int32 /*OldLevel*/, int32 /*NewLevel*/);
FOnDOAbilityLevelChanged OnAbilityLevelChanged;
```

- 在 `SetDOAbilityLevel` 成功改级后 `OnAbilityLevelChanged.Broadcast(AbilityId, OldLevel, NewLevel)`。
- 客户端监听该委托（或监听 GAS 的 `AbilitySpecDirtiedCallbacks`），在本地 `BroadcastMessage(Message.UI.Skill.LevelChanged, Payload)` 让 UI 刷新。
- **不需要** Crunch 那种手写 Client RPC：等级值已由 `Mixed` 复制同步，Message 总线只负责"通知表现层刷新"，符合 `Docs/04` 的边界（Message 不承载请求）。

### 4.5 升级经济：技能点（SkillPoints）

当前**不存在**技能点属性，需新增（执行阶段单独规划，本方案先定接口）：

- 建议新增属性集 `UDOSkillSet`（或并入 `UDOCombatSet`，视属性归属约定），含：
  - `SkillPoints`（可用技能点，Replicated）— 升级消耗从这里扣。
  - `MaxSkillPointsPerLevel` 等可扩展字段。
- 技能点来源（升级获得 / 任务奖励等）由服务器决定，不在本方案范围，先留 `TODO`。
- 升级消耗计算：`Cost = NewLevel - OldLevel`（每级 1 点；若后续需非线性消耗，从 `SkillTreeComponent`/配置读取）。

> 占位说明：技能点系统是独立的资源模块，本方案只定义"升级时从技能点扣 `NewLevel-OldLevel`"这一约束，具体经济（获得、上限、重置返还）后续单独设计。

### 4.6 技能树前置 / 消耗管理（SkillTreeComponent，占位）

`DOAbilitySet.h:39` 注释已把前置/消耗归到 `SkillTreeComponent`，该组件当前未实现。本方案：

- 在 `Server_RequestAbilityLevel_Implementation` 中预留 `SkillTreeComponent` 校验钩子（如 `CanUpgrade(AbilityId, NewLevel)`）。
- 组件职责（执行阶段再细化）：持有每个 `AbilityId` 的前置条件（依赖技能+等级、职业 Tag、角色等级）、每级消耗公式、是否已解锁。
- 当前无该组件时，校验退化为"仅 MaxLevel + 技能点"两项。

## 五、数据流（联机）

```text
[客户端] 玩家在升级菜单点"升级 X 到 Lv.N"
  -> 调用本地 ASC->Server_RequestAbilityLevel(X, N)        // Server RPC 跨网络
      |
[服务端] Server_RequestAbilityLevel_Implementation
  -> 校验: 已授予? 区间? 有变化? 技能点够? 角色/树前置?
  -> 通过: 扣 SkillPoints (N - Old)
           -> SetDOAbilityLevel(X, N)
                -> Spec->Level = N; MarkAbilitySpecDirty   // GAS Mixed 复制 Level 到客户端
                -> OnAbilityLevelChanged.Broadcast(X, Old, N)
  -> 失败: 忽略（可选 BroadcastMessage(Message.UI.Skill.UpgradeFailed)）

[客户端] 收到 Spec.Level 复制 (Mixed)
  -> 监听 OnAbilityLevelChanged / AbilitySpecDirtiedCallbacks
  -> BroadcastMessage(Message.UI.Skill.LevelChanged, {AbilityId, Old, New})
  -> UI Widget 监听 Message 刷新节点状态 + 剩余技能点
```

要点：
- **客户端从不直接改 Level**，所有权威判定在服务端。
- **等级同步零额外网络代码**（`MarkAbilitySpecDirty` 已搞定）。
- **UI 刷新走 Message 总线**（表现层解耦），不发 Client RPC。

## 六、与 Crunch 的差异对照

| 维度 | Crunch | DragonOath（本方案） |
|---|---|---|
| 标识 | `ECAbilityInputID` 枚举 | `FGameplayTag AbilityId`（合规） |
| 请求入口 | `Server_UpgradeAbilityWithID` RPC | `Server_RequestAbilityLevel` RPC（同思路） |
| 校验位置 | `_Validate` + 实现（升级点/等级） | `_Validate` + 实现（技能点/MaxLevel/前置） |
| 等级同步 | `MarkAbilitySpecDirty` + 手写 `Client_AbilitySpecLevelUpdated` | 仅 `MarkAbilitySpecDirty`，UI 经 Message 总线刷新 |
| 升级经济 | `UpgradePoint` 属性 | `SkillPoints`（新增，占位） |
| UI 触发 | 直接调 RPC | 经 RPC → 服务端 → 复制 → Message → UI |

## 七、UI 需要做什么（详细，暂不实现）

> 本节为 UI 开发指南，本方案不写任何 Widget/C++ UI 代码。后续实现时按此落地。

### 7.1 需要的界面
- **技能树 / 升级面板**（`WBP_SkillTree` 或 `WBP_UpgradeMenu`）：展示当前职业全部 `FDOAbilityGrant`（AbilityId、当前等级、MaxLevel、前置、每级消耗）。
- **技能点显示**：面板顶部展示 `可用技能点 / 总点数`。
- **单个技能节点**（`WBP_SkillNode`）：显示图标、名称、当前/最大等级、升级按钮（可点条件：未达 Max、技能点够、前置满足）。

### 7.2 UI 行为约束
- **升级必须由客户端发请求，不能本地改数值**：点升级按钮 → 取本地 `UDOAbilitySystemComponent` → 调 `Server_RequestAbilityLevel(AbilityId, CurrentLevel+1)`。UI 本地**不要**自行加等级或扣点，一切以服务端复制结果为准。
- **客户端何时能"预读"可升级性**：UI 可本地读取 `FDOAbilityGrant`（静态配置，含 MaxLevel）与本地 `SkillPoints` 副本做按钮可用态预判，但最终能否升由服务端裁决；若服务端拒绝，UI 会在收到复制/失败时回滚显示。

### 7.3 UI 监听的数据源
| 数据 | 来源 | UI 用法 |
|---|---|---|
| 技能当前等级 | GAS `Spec.Level`（Mixed 复制） | 节点等级显示 |
| 等级变化事件 | `Message.UI.Skill.LevelChanged` Payload: `{AbilityId, OldLevel, NewLevel}` | 刷新节点、播放升级特效 |
| 可用技能点 | `SkillPoints` 属性复制 / `Message.UI.Skill.PointsChanged` | 顶部点数、按钮可用态 |
| 升级失败（可选） | `Message.UI.Skill.UpgradeFailed` Payload: `{AbilityId, Reason}` | Toast 提示"技能点不足/前置未满足" |
| 前置状态 | 本地读 `SkillTreeComponent` 或配置 | 灰化未解锁节点、显示前置链 |

### 7.4 UI 初始化流程（建议）
1. `NativeConstruct`：注册 `Message.UI.Skill.LevelChanged`、`Message.UI.Skill.PointsChanged` 监听；从本地 ASC 拉取各 `AbilityId` 的当前 `Spec.Level` 与 `SkillPoints` 初始化显示。
2. 打开面板时：遍历职业 `UDOAbilitySet::GrantedAbilities`，为每个 `FDOAbilityGrant` 建节点，绑定升级按钮 `OnClicked -> Server_RequestAbilityLevel`。
3. `NativeDestruct`：`Unregister` 所有 `FGameplayMessageListenerHandle`。

### 7.5 Message Payload 约定（按 `Docs/04`）
```cpp
USTRUCT(BlueprintType)
struct FDOSkillLevelChangedMessage
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) FGameplayTag AbilityId;
    UPROPERTY(BlueprintReadOnly) int32 OldLevel = 0;
    UPROPERTY(BlueprintReadOnly) int32 NewLevel = 0;
};
// 频道: Message.UI.Skill.LevelChanged（精确匹配）
```

## 八、联机验证项（执行后 PIE 自测）

1. **PIE Listen Server + 1 Client**：客户端打开升级面板，点升级 → 服务端 `_Implementation` 校验通过 → `Spec.Level` 经 Mixed 复制回客户端 → `Message.UI.Skill.LevelChanged` 触发 → UI 刷新。
2. **防作弊**：在客户端直接调用 `SetDOAbilityLevel`（绕过 RPC）应被服务端角色检查拒绝（返回 false，不改 Level）。
3. **技能点扣除**：升级成功后 `SkillPoints` 减少 `NewLevel - OldLevel`；点数不足时升级被拒、点数不变。
4. **MaxLevel 边界**：`NewLevel > MaxLevel` 的请求被 `_Validate`/实现拒绝。
5. **0↔1 副作用保持**：升级到 1 时 `OnSpawn` 被动自动激活；降到 0 时取消运行中的技能（现有逻辑不破坏）。
6. **无预测回滚风险**：升级是纯服务端权威项，不进 `LocalPredicted` 预测路径，无需处理回滚。
