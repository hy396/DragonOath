# 技能动态冷却与冷却缩减方案

> 阶段：M2 战斗闭环期。为技能层补齐「运行时可配置冷却」与「冷却缩减 / 重置」两项机制。
> 关联：`Docs/方案/技能按键布局与职业技能规划.md`、`Docs/ProfessionAbilityConfigDesign.md`、`Docs/06_Combat_Attribute_Design.md`、`AGENTS.md`「GAS 约定」、`Source/DragonOath/AbilitySystem/`。
> 来源：桌面 `GA动态技能.pdf`（`Xh` 前缀的参考实现）。本方案已按 DragonOath（`DO` 前缀 + GAS 三层 + 引用 `DOGameplayTag` 变量）的规范重写，并**修正/优化了参考实现中绕开 GAS 原生冷却管线的部分**。

---

## 0. 背景与目标

### 现状

- 现有技能（`Dash` / `DashAttack` / `NormalAttack`）激活时都走 `CommitAbility`，冷却完全依赖 GAS 原生的 `CooldownGameplayEffectClass`（固定时长，写死在 GE 蓝图里）。
- 冷却时长要改就得改 GE 资产；同一技能不同等级 / 天赋想要不同 CD，只能拆多个 GE，无法「一个技能一个 CD 数值」在技能蓝图属性面板直接填。
- 没有任何「缩短 / 重置 / 延长其它技能冷却」的能力——而技能规划里已经预留了这类设计（大招 CD 长、招架 / 战意类功能技需要联动 CD）。

### 目标

| 子功能                       | 说明                                                                                            | 典型用途                                                           |
| ---------------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| **A. 动态冷却**        | 冷却时长由技能蓝图属性面板直接填（`SetByCaller` 注入到共享冷却 GE），一个 GE 资产服务所有技能 | 每个技能一个 CD 数值；未来可按等级 / 天赋缩放                      |
| **B. 冷却缩减 / 重置** | 运行时按 Tag 查找已激活的冷却 GE，增减其剩余时长（可一次改多个）                                | 命中回 CD、连招重置、"战意"减 CD、处决重置大招、Debuff 延长敌方 CD |

两项都必须满足项目的**联机优先**要求：冷却的施加与修改均**服务器权威**，通过 GAS 预测 / FastArray 复制到客户端。

## 1. 与现状的关系：先理解 GAS 原生冷却

`UGameplayAbility` 已内建一套冷却管线，本方案是**接管**它而不是另起炉灶：

```
CommitAbility
  └─ CommitAbilityCooldown → ApplyCooldown()            // 施加冷却 GE
CanActivateAbility / CheckCooldown
  └─ GetCooldownTags() → ASC.HasAnyMatchingGameplayTags // 有冷却 Tag 则拒绝激活
UI 查询
  └─ GetCooldownTimeRemaining / ...AndDuration          // 读剩余时长
```

已在引擎中确认的关键 API（`D:\UE_5.8\...\GameplayAbilities`）：

- `UGameplayAbility::GetCooldownGameplayEffect()` — 默认返回 `CooldownGameplayEffectClass` 的 CDO。
- `UGameplayAbility::ApplyCooldown(...)`（`virtual`）— 默认 `ApplyGameplayEffectToOwner(CooldownGE)`，**不带 SetByCaller**，这正是我们要覆写的点。
- `UGameplayAbility::GetCooldownTags()`（`virtual`）— `CheckCooldown` 据此判断是否在冷却中。
- `FActiveGameplayEffectsContainer::ModifyActiveEffectStartTime(Handle, StartTimeDiff)` — `StartWorldTime += StartTimeDiff` 后 `CheckDuration` + `MarkItemDirty`（复制）。

> **重要结论（决定为何优于 PDF 写法）**：只要冷却仍走原生 `ApplyCooldown` + `GetCooldownTags`，`GetCooldownTimeRemaining` 等 UI 查询就能正常工作。PDF 里另写 `ApplyCooldownGE` 并用 `ActivationBlockedTags` 挡激活，等于绕开 `CheckCooldown`，导致 UI 的技能冷却查询失效、且每个技能要单独做一个冷却 GE 蓝图。本方案改为**共享一个冷却 GE + 覆写 `ApplyCooldown` / `GetCooldownTags`**。

## 3. GameplayTag 规划

在 `Source/DragonOath/AbilitySystem/Core/DOGameplayTag.h` / `.cpp` 新增（沿用现有 `UE_DECLARE_GAMEPLAY_TAG_EXTERN` / `UE_DEFINE_GAMEPLAY_TAG_COMMENT`，注释中文）：

```cpp
// DOGameplayTag.h —— SetByCaller 数据传递标签 namespace Data 内
// 冷却时长（秒），通过 SetByCaller 传给共享冷却 GE
DRAGONOATH_API UE_DECLARE_GAMEPLAY_TAG_EXTERN(CooldownDuration);

// DOGameplayTag.h —— 新增 namespace Cooldown（每技能一个冷却标识）
namespace Cooldown
{
    namespace DragonFighter
    {
        DRAGONOATH_API UE_DECLARE_GAMEPLAY_TAG_EXTERN(DashStrike);
        DRAGONOATH_API UE_DECLARE_GAMEPLAY_TAG_EXTERN(DragonFury);   // 大招，长 CD
        // ...按 技能按键布局与职业技能规划.md 逐个补
    }
    // Mage / Archer 同理
}
```

`Cooldown.<职业>.<技能名>` 这一个 Tag **三重用途**：

1. `ApplyCooldown` 时作为 **DynamicGrantedTags** 授予 Owner → ASC 持有该 Tag。
2. `GetCooldownTags()` 返回它 → `CheckCooldown` 靠它拒绝重复激活。
3. `ApplyCooldown` 时作为 **DynamicAssetTags** 注入 → 子功能 B 用 `EffectTagQuery` 才能按 Tag 查到这个冷却 GE。

> **已验证的坑（务必按此实现）**：`FGameplayEffectQuery::EffectTagQuery` 只匹配 **AssetTags**（`Spec.GetAllAssetTags`），不匹配 GrantedTags。所以「授予 Owner 用于挡激活」和「按 Tag 查询用于缩减」是两套 Tag 集合——同一个 `CooldownTag` 必须**同时**加进 `DynamicGrantedTags` 和 `DynamicAssetTags`，缺一不可。

命名沿用现有 `Ability.<职业>.<技能名>` 的 AbilityId 风格，冷却 Tag 用平行的 `Cooldown.<职业>.<技能名>`，一一对应，便于 UI / 缩减系统按职业批量处理。

## 4. 子功能 A：动态冷却（SetByCaller 时长）

### 4.1 共享冷却 GE 资产（`GE_Cooldown_Dynamic`）

一个蓝图 GE，全项目技能复用：

- **Duration Policy** = `Has Duration`。
- **Duration Magnitude** = `Set By Caller`，Data Tag = `Data.CooldownDuration`。
- 不配任何静态 Granted / Asset Tag（每技能的 `CooldownTag` 在运行时动态注入，见下）。

> 与 PDF 差异：PDF 给每个技能做一个冷却 GE，并在 GE 上静态配 Target Tag + Asset Tag。本方案只做**一个**共享 GE，Tag 运行时注入，资产更少、扩展性更好。

### 4.2 `UDOGameplayAbility` 改动

复用基类已有的 `CooldownGameplayEffectClass`（设计师在技能蓝图里指向 `GE_Cooldown_Dynamic`），只新增两个字段 + 覆写两个虚函数：

```cpp
// DOGameplayAbility.h（protected 区）
protected:
    // 本技能的冷却标识 Tag（Cooldown.<职业>.<技能名>）。三重用途见方案 §3。
    UPROPERTY(EditDefaultsOnly, BlueprintReadOnly, Category="DO|Cooldown", meta=(Categories="Cooldown"))
    FGameplayTag CooldownTag;

    // 冷却时长（秒）。0 表示无冷却。设计师直接在技能蓝图属性面板填。
    UPROPERTY(EditDefaultsOnly, BlueprintReadOnly, Category="DO|Cooldown")
    float CooldownDuration = 0.0f;

    //~ 接管 GAS 原生冷却管线
    virtual const FGameplayTagContainer* GetCooldownTags() const override;
    virtual void ApplyCooldown(const FGameplayAbilitySpecHandle Handle,
        const FGameplayAbilityActorInfo* ActorInfo,
        const FGameplayAbilityActivationInfo ActivationInfo) const override;

private:
    // GetCooldownTags 的返回缓冲：父类冷却 Tag ∪ 本技能 CooldownTag。
    UPROPERTY(Transient)
    mutable FGameplayTagContainer TempCooldownTags;
```

```cpp
// DOGameplayAbility.cpp

const FGameplayTagContainer* UDOGameplayAbility::GetCooldownTags() const
{
    // 合并父类冷却 GE 自带 Tag（一般为空）与本技能动态 CooldownTag，
    // 供 CheckCooldown 判断"是否处于冷却"。
    TempCooldownTags.Reset();
    if (const FGameplayTagContainer* ParentTags = Super::GetCooldownTags())
    {
        TempCooldownTags.AppendTags(*ParentTags);
    }
    if (CooldownTag.IsValid())
    {
        TempCooldownTags.AddTag(CooldownTag);
    }
    return &TempCooldownTags;
}

void UDOGameplayAbility::ApplyCooldown(const FGameplayAbilitySpecHandle Handle,
    const FGameplayAbilityActorInfo* ActorInfo,
    const FGameplayAbilityActivationInfo ActivationInfo) const
{
    UGameplayEffect* CooldownGE = GetCooldownGameplayEffect(); // = CooldownGameplayEffectClass CDO
    if (!CooldownGE || CooldownDuration <= 0.0f || !CooldownTag.IsValid())
    {
        return; // 无冷却配置则直接跳过
    }

    FGameplayEffectSpecHandle Spec = MakeOutgoingGameplayEffectSpec(CooldownGE->GetClass(), GetAbilityLevel());
    if (!Spec.IsValid())
    {
        return;
    }

    // ① 授予 Owner：用于 GetCooldownTags/CheckCooldown 挡重复激活
    Spec.Data->DynamicGrantedTags.AddTag(CooldownTag);
    // ② 资产 Tag：用于子功能 B 的 EffectTagQuery 按 Tag 查询（GrantedTags 查不到）
    Spec.Data->AddDynamicAssetTag(CooldownTag);
    // ③ SetByCaller 注入冷却时长
    Spec.Data->SetSetByCallerMagnitude(DragonOathGameplayTags::Data::CooldownDuration, CooldownDuration);

    ApplyGameplayEffectSpecToOwner(Handle, ActorInfo, ActivationInfo, Spec);
}
```

### 4.3 现有技能接入

- 现有 `Dash` / `NormalAttack` 等把基类 `CooldownGameplayEffectClass` 指向 `GE_Cooldown_Dynamic`，填 `CooldownTag` + `CooldownDuration` 即可；`CommitAbility` 无需改动（内部会调到覆写后的 `ApplyCooldown`）。
- `CooldownDuration = 0` 的技能（普攻）保持无冷却，覆写里已 early-return。

## 5. 子功能 B：冷却缩减 / 重置

### 5.1 `UDOAbilitySystemComponent::AdjustEffectRemainingDuration`

改一个已激活 GE 的剩余时长（冷却缩减 / 延长的底层原子操作）。必须放在 ASC 子类，因为 `ActiveGameplayEffects`（`FActiveGameplayEffectsContainer`）是 ASC 的 protected 成员。

```cpp
// DOAbilitySystemComponent.h（public）
// 调整某个已激活 GE 的剩余时长。DurationDelta：正=延长，负=缩短。
// 仅服务器权威调用；通过 FastArray 复制到客户端。
UFUNCTION(BlueprintCallable, Category="DO|Cooldown")
void AdjustEffectRemainingDuration(FActiveGameplayEffectHandle Handle, float DurationDelta);
```

```cpp
// DOAbilitySystemComponent.cpp
void UDOAbilitySystemComponent::AdjustEffectRemainingDuration(FActiveGameplayEffectHandle Handle, float DurationDelta)
{
    if (!Handle.IsValid())
    {
        return;
    }

    const FActiveGameplayEffect* ActiveGE = GetActiveGameplayEffect(Handle);
    if (!ActiveGE)
    {
        return;
    }

    // 只处理"有限时长"的 GE，跳过瞬时 / 无限
    const float Duration = ActiveGE->Spec.GetDuration();
    if (Duration == FGameplayEffectConstants::INFINITE_DURATION ||
        Duration == FGameplayEffectConstants::INSTANT_APPLICATION)
    {
        return;
    }

    // ModifyActiveEffectStartTime 语义（已核对引擎实现）：
    //   StartWorldTime += DurationDelta → 剩余时长同向变化。
    //   正值 = 开始时间后移 = 剩余变长（延长 CD）；负值 = 剩余变短（缩短 CD）。
    // 内部会 CheckDuration（可能立即到期移除）+ MarkItemDirty（复制）。
    ActiveGameplayEffects.ModifyActiveEffectStartTime(Handle, DurationDelta);
}
```

> **符号约定已核对引擎**：`ModifyActiveEffectStartTime` 做 `StartWorldTime += StartTimeDiff`。剩余 = `Duration - (Now - StartWorldTime)`，故 **正值延长、负值缩短**，与 PDF 注释一致。缩短到 ≤0 时 `CheckDuration` 会立刻移除该冷却 GE = 冷却重置。

### 5.2 批量入口：按 Tag 增减多个冷却

一次修改多个冷却 GE。**推荐用普通 `BlueprintCallable`，不用 `UAbilityTask`**——该操作是同步的（无 latent 行为），`UAbilityTask` 只是为了得到一个蓝图节点，收益不抵样板成本。

```cpp
// DOAbilitySystemComponent.h（public）
// 按 Tag 批量调整冷却剩余时长。Key=冷却 GE 的资产 Tag（Cooldown.*），Value=时长增减（正延长/负缩短）。
// 仅服务器权威调用。
UFUNCTION(BlueprintCallable, Category="DO|Cooldown")
void AdjustCooldownsByTag(const TMap<FGameplayTag, float>& CooldownAdjustments);
```

```cpp
// DOAbilitySystemComponent.cpp
void UDOAbilitySystemComponent::AdjustCooldownsByTag(const TMap<FGameplayTag, float>& CooldownAdjustments)
{
    if (!IsOwnerActorAuthoritative())
    {
        return; // 权威端才改冷却，结果经复制下发
    }

    for (const TPair<FGameplayTag, float>& Pair : CooldownAdjustments)
    {
        FGameplayEffectQuery Query;
        Query.EffectTagQuery = FGameplayTagQuery::MakeQuery_MatchTag(Pair.Key);

        for (const FActiveGameplayEffectHandle& Handle : GetActiveEffects(Query))
        {
            AdjustEffectRemainingDuration(Handle, Pair.Value);
        }
    }
}
```

- 从技能蓝图（`ActivateAbility` 内）调用即可：命中回 CD、连招重置、"战意"减 CD 等，全在职业技能蓝图里配 `TMap`。
- **可选**：若确实希望在图里得到 `Async Task` 风格节点，可另做一个薄封装 `UAbilityTask`（`UDOAT_AdjustCooldowns`），`Activate()` 里调 ASC 的 `AdjustCooldownsByTag` 后立即 `EndTask()`。默认不做，等有蓝图人力需求再加，避免样板。

## 6. 网络与权威（联机优先，务必遵守）

- **冷却施加（A）**：`ApplyCooldown` 由 `CommitAbility` 触发。DO 技能默认 `InstancedPerActor + LocalPredicted`，冷却 GE 会在客户端预测、服务器确认，天然联机安全，无需额外处理。
- **冷却缩减 / 重置（B）**：`ModifyActiveEffectStartTime` 会 `MarkItemDirty` 从服务器复制到客户端。因此 `AdjustCooldownsByTag` / `AdjustEffectRemainingDuration` **只能在服务器权威端执行**（已用 `IsOwnerActorAuthoritative()` 守卫）。
  - 若缩减逻辑写在 `LocalPredicted` 技能里，客户端那一侧不要直接改（会与复制打架）——用 `if (K2_HasAuthority())` 或把缩减放进技能的服务器阶段 / Server RPC。蓝图待办里会明确标注。
- **怪物 ASC**：挂在 `ADOCharacter` 自身，同样服务器权威，逻辑一致。

## 7. 与 PDF 参考实现的差异与改进

| 维度         | PDF（`Xh`）写法                                             | 本方案（`DO`）                                                                     | 理由                                                   |
| ------------ | ------------------------------------------------------------- | ------------------------------------------------------------------------------------ | ------------------------------------------------------ |
| 冷却 GE 数量 | 每技能一个冷却 GE 蓝图                                        | 全项目共享一个`GE_Cooldown_Dynamic`                                                | 资产更少，冷却 Tag 运行时注入                          |
| 施加方式     | 自写`ApplyCooldownGE(float)`，绕过 `CommitAbility`        | 覆写原生`ApplyCooldown` + `GetCooldownTags`                                      | 保留`GetCooldownTimeRemaining` 等 UI 查询            |
| 挡重复激活   | `ActivationBlockedTags` 手配相同 Tag                        | 走原生`CheckCooldown`（`GetCooldownTags`）                                       | 单一事实来源，UI/逻辑一致                              |
| 查询用 Tag   | 资产 Tag（GE 上静态配）                                       | 运行时`AddDynamicAssetTag`                                                         | 配合共享 GE；已核对`EffectTagQuery` 仅匹配 AssetTags |
| 批量入口     | `UAbilityTask`（同步却用异步节点）                          | `BlueprintCallable`（AbilityTask 可选）                                            | 同步操作无需 latent 样板                               |
| 权威         | 未显式强调                                                    | 显式`IsOwnerActorAuthoritative` 守卫                                               | 联机优先项目必须                                       |
| Tag / 命名   | `Ability.CoolingDuration`、`Ability.Weapon.Spear.ThrowCD` | `Data.CooldownDuration`、`Cooldown.<职业>.<技能名>`，引用 `DOGameplayTag` 变量 | 遵守项目「不手写 RequestGameplayTag 字符串」约定       |

保留自 PDF 的正确核心：`SetByCaller` 注入冷却时长、`ModifyActiveEffectStartTime` 调整剩余时长、`TMap<Tag,float>` 一次改多个、正负号语义。

## 10. 测试与验收

- **A 动态冷却**：不同 `CooldownDuration` 的技能冷却时长正确；冷却期间无法重复激活；UI 冷却条与实际一致。
- **B 缩减**：负值缩短、正值延长；缩到 ≤0 时冷却 GE 立即移除（可再次激活）；`TMap` 一次改多个 Tag 生效。
- **边界**：无限 / 瞬时 GE 不受影响；无效 Handle / 空 Tag 安全返回；`CooldownDuration=0` 技能无冷却。
- **联机**：客户端预测冷却与服务器一致；缩减仅服务器执行并正确复制；非权威端调用被守卫拦截。
